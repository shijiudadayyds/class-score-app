# 班级加减分助手：重构与升级路线图

## 当前判断

这个项目已经具备完整可用的桌面应用能力，核心体验已经成型：

- Electron 桌面端，支持安装包和便携版打包
- 班级面板、学生、小组、积分、历史记录、排行榜
- 名单导入、备份导入导出、Excel 汇总
- 随机点名、计时器、倒计时、大屏展示
- 宠物家园、宠物商城、积分商城
- 悬浮窗联动

但当前最主要的工程风险，不在主进程，而在渲染层：

- `src/renderer.js` 体量约 `9000+` 行
- 业务、状态、渲染、弹窗、工具函数高度耦合
- 多个函数存在重复定义，后定义覆盖前定义
- `src/renderer-overrides.js` 通过后加载方式继续覆盖逻辑

这意味着项目现在最值得优先处理的不是“再堆一个大功能”，而是先把渲染层拆出边界。

## 第一优先级：先拆渲染层

### 为什么先拆这里

这一层同时负责：

- 应用状态
- 保存与恢复
- 撤销 / 重做
- 学生和小组积分逻辑
- 随机点名
- 排行榜
- 倒计时 / 秒表
- 弹窗系统
- 宠物和商城系统

继续在当前结构上加功能，短期能做，长期会越来越难维护。最直观的信号是同名函数重复定义，例如：

- `renderPetHomeModal`
- `openPetHomeModal`
- `renderPetAvatarMarkup`
- `renderShopModal`
- `applyDeltaToStudents`

这些重复定义会导致：

- 改动时很难确认最终生效的是哪一版
- 排查 bug 时定位成本高
- 新功能容易误伤旧逻辑
- 后续接手的人理解成本很高

## 推荐拆分结果

## 实施顺序

为了降低风险，不建议一次性重写。建议分四段推进。

### 阶段 1：抽基础能力

先从最稳定、最通用的部分抽离：

- `appState`、`selectedStudentId`、`checkedStudentIds`
- `serializeState`
- `createSnapshot`
- `undoLastAction` / `redoLastAction`
- `commitState`
- `scheduleSave`
- `syncWidgetState`
- `openModal`
- `showToast`

这一阶段的目标不是改变功能，而是把“所有功能都依赖的地基”单独收拢。

### 阶段 2：抽课堂主流程

第二阶段优先抽和主业务最相关的模块：

- 学生增删改查
- 小组管理
- 加减分动作
- 历史记录
- 排行榜
- 随机点名

抽完这一步，项目核心价值就已经从“大单文件脚本”变成“主流程模块化”。

### 阶段 3：抽宠物与商城

宠物和商城建议单独作为完整域处理：

- 宠物数据结构
- 宠物成长、喂养、孵化、技能
- 商城道具
- 宠物商城与积分商城的区分
- 战斗逻辑

这是目前文件里最重、重复最多、最适合独立隔离的一块。

### 阶段 4：移除覆盖式补丁

最后处理 `renderer-overrides.js`：

- 逐个确认被覆盖的函数已经被归入正式模块
- 迁移完成后删除重复定义
- 最终让 `renderer-overrides.js` 退场

完成这一阶段，项目维护难度会明显下降。

## 立即可做的 3 个重构任务

如果只做最小投入、最大收益的动作，建议按下面顺序执行：

1. 提取状态管理与撤销重做链路
2. 提取弹窗与 Toast 能力，减少重复事件绑定
3. 提取宠物 / 商城模块，消除重复函数定义

## 功能升级建议

下面这些功能比“继续堆宠物玩法”更值得优先做。

### A 级：强烈推荐

#### 1. 导入预览与字段映射

当前名单导入已经支持多格式，但还可以更稳：

- 导入前预览识别结果
- 手动指定姓名列、学号列、座号列、小组列
- 显示“将新增 / 将跳过 / 将补全”的数量
- 允许选择“覆盖资料”或“仅补全空字段”

价值：

- 明显减少误导入
- 降低老师使用门槛
- 是最容易提升体验的一项

#### 2. 课堂规则模板

允许每个面板一键套用规则模板：

- 加分项模板
- 减分项模板
- 步进分值
- 小组设定
- 商城设定

价值：

- 不同班级 / 不同老师切换更方便
- 减少重复配置
- 很适合后续做“模板导出 / 导入”

#### 3. 公平随机点名

当前随机点名可以继续做得更实用：

- 本轮不重复
- 最近 N 次不重复
- 优先未被点到的学生
- 按小组轮流点名
- 过滤缺勤学生

价值：

- 课堂现场体验提升很明显
- 逻辑清晰，开发风险低

### B 级：推荐

#### 4. 座位表模式

增加班级座位布局视图：

- 按座位点名
- 按座位批量加减分
- 一眼看到缺勤和分数状态

价值：

- 更符合真实课堂场景
- 比纯名单模式更直观

#### 5. 周报 / 月报

除了导出总分，还可以输出：

- 学生进步榜
- 学生波动榜
- 小组表现趋势
- 加分 / 扣分原因统计

价值：

- 更适合班主任长期使用
- 也利于家校沟通

#### 6. 自动保护点

在关键操作前自动创建安全快照：

- 导入名单前后
- 批量加分前
- 清空面板前
- 重置分数前

价值：

- 当前项目已经有保护点基础
- 这是低风险、高收益的增强

### C 级：可选增强

#### 7. 兑换审核流

当前积分商城可以继续升级为：

- 学生发起兑换
- 老师审核通过
- 审核记录可回看

更适合真实班级管理。

#### 8. 成长趋势图

给学生和小组展示最近一段时间的曲线变化，而不只是静态总分。

#### 9. 极简大屏模式

增加纯展示视图，只保留：

- 排行榜
- 倒计时
- 当前面板信息
- 随机点名结果

这样投屏时会更干净。

## 建议暂缓的事项

下面这些不是不能做，而是现在不该排在前面：

- 继续扩大量宠物分支玩法
- 再往 `renderer.js` 里直接堆大块新逻辑
- 在没有拆模块前引入更多覆盖脚本

原因很简单：这些都会继续放大当前结构问题。

## 推荐路线

如果按“投入产出比”排序，建议路线如下：

1. 先做渲染层拆分，尤其是状态层和宠物 / 商城模块
2. 然后做“导入预览 + 字段映射”
3. 再做“公平随机点名”
4. 接着做“课堂规则模板”
5. 最后再考虑座位表、报表和审核流

## 结论

这个项目目前已经不是“缺功能”，而是“缺边界”。

最值得先做的事，不是继续把功能做大，而是先让代码结构跟上功能规模。只要把渲染层拆开，后面的导入优化、课堂模板、公平点名、座位表、周报这些功能，都会变得更好做、更稳、更容易长期维护。
